## 二、实用性审查 — 发现问题

### 🔴 P0 — 致命问题（员工无法信任这个网站）

#### P0-1：大量"假数据"混在展示中，用户无法分辨真假

**现状**：

| 数据类型 | 实际性质 | 问题 |
|----------|----------|------|
| 行业指数面板（半导体=4856.32 / 汽车=2892.45 / AI=4521.89） | **静态硬编码**，永不变化 | 完全不可信 |
| 大盘图表（Chart.js） | **三角函数生成**，`Math.sin(i)/Math.cos(i)` | 骗人图表 |
| "晶圆厂产能 92%" / "芯片交期 40周" | 静态写死的数字 | 没有更新 |
| "新能源车销量 285万" / "人形机器人出货量 12.5万" | 静态写死 | 数据来源标注是假的 |

**后果**：
- 员工看几天后就会发现"数据从来没变过" → 彻底失去信任
- Chart.js 的图表每天看起来都是同一个正弦曲线 → 明显看出是生成数据
- 一旦失去信任，整个网站沦为"摆设"，没人会再看

**建议**（按可行性排序）：

| 方案 | 工作量 | 效果 |
|------|--------|------|
| **方案A**：所有静态数据添加明确的 `[静态参考]` 标签，与实时数据视觉区分 | 1天 | 诚实标注，至少不会误导 |
| **方案B**：用东方财富 JSONP 抓取指数实时值（已证实可行，可复用股票抓取模块） | 3天 | 真正实现数据实时更新 |
| **方案C**：移除不可信的静态数据面板，只展示从 RSS/API 实时获取的数据 | 0.5天 | 数据量减少但信息可信任 |
| **方案D（推荐）**：A+B 组合 — 先诚实标注，再逐步接入实时数据 | 1+3天 | 短期可信 + 长期可用 |

#### P0-2：兜底模板数据与真实新闻无法区分

**现状**：
- 当 RSS 抓取失败时，系统自动回退到 `NEWS_TEMPLATES` / `HOTSPOT_TEMPLATES`
- 模板数据看起来和真实新闻一模一样（有标题、摘要、时间戳）
- 用户**完全不知道**看到的是兜底的假新闻还是真实抓取的新闻

示例（来自 `staticData.ts`）：
```typescript
{ title: '台积电 N2P 制程试产进展顺利', summary: '良率有望达预期目标', ... }
```

**后果**：
- 用户可能把 3 天前的兜底数据当成今日新闻，做出错误判断
- 来源标注缺失或不准确

**建议**：
- 所有非实时数据必须标注 `[缓存]` / `[离线]` 标签（带颜色区分）
- 在 Header 的 LIVE 指示器旁增加**数据新鲜度指示器**（如 `上次更新: 3分钟前` vs `⚠️ 离线数据`)
- 兜底数据在标题前加 `[参考]` 前缀

#### P0-3：无"原文链接"——只能看摘要，不能深入阅读

**现状**：
- 新闻卡片只显示标题 + 60 字摘要
- **没有任何链接或点击交互**
- 用户看到感兴趣的新闻后，必须自己手动复制标题去搜索引擎查询

**后果**：
- 这是"监控系统"最大硬伤 — 监控的目的是发现+深入，但现在只有发现没有深入
- 极其低效：用户必须切换工具才能确认信息
- 竞争对手情报的价值在于"快速获取原文"——当前完全做不到

**建议**：
```typescript
// 在每条新闻卡片上增加来源链接
<div class="news-card" onclick="window.open('${item.link}', '_blank')">
  ${item.title}
  <span class="source-link">📎 阅读原文</span>
</div>
```
- rss2json API 返回的每个 item 都包含 `link` 字段，直接可用
- 如果来源是兜底模板（无真实链接），则隐藏链接按钮

### 🟡 P1 — 重要问题（影响日常使用体验）

#### P1-1：欧冶自身的核心业务线覆盖不足

**现状**：5 个 Tab 分别是全局/半导体/智能汽车/机器人/AI

**缺失的关键板块**：

| 欧冶核心业务 | 当前覆盖情况 | 需要追踪什么 |
|-------------|-------------|-------------|
| **端侧 AI（感算一体）** | ❌ 完全无覆盖 | 边缘计算芯片、端侧推理、NPU IP、低功耗 AI SoC |
| **车载芯片** | ⚠️ 混在"智能汽车"里，不是独立视角 | 智驾芯片出货、竞品对比、车企定点 |
| **双目深度引擎** | ❌ 完全无覆盖 | 立体视觉、深度传感器、3D 感知方案 |
| **具身智能** | ⚠️ 混在"机器人"里 | 人形机器人芯片、感知-决策-控制链路 |
| **竞争对手追踪** | ⚠️ 仅有通用竞争对手（NVDA/QCOM/AMD） | 需要地平线/黑芝麻/寒武纪/芯驰/奕斯伟等中国芯片公司 |
| **产业政策** | ⚠️ 仅有工信部一个源 | 需要科技部/发改委/工信部/地方半导体专项政策 |
| **专利与 IP** | ❌ 完全无覆盖 | 芯片专利动态、标准必要专利、侵权诉讼 |

**建议**：
- **增加「端侧 AI」Tab**：这是欧冶的核心差异化竞争点
- **竞争对手从 6 家扩展到 15-20 家**：覆盖中国芯片赛道的真实对手
- **增加「车规芯片」子面板**：定点信息、客户合作进展、AEC-Q100 认证动态
- **增加「产业政策」独立 Tab**：深圳半导体专项、国家重大科技专项、税收优惠

#### P1-2：RSS 源几乎全是英文，中国半导体产业视角缺失

**现状**：18 个 RSS 源中，**仅 1 个中文源**（36氪），其余全是英文

| 类型 | 英文源 | 中文源 |
|------|--------|--------|
| 半导体 | EE Times, Semi Engineering, Semi Digest, SemiAnalysis, Semiconductor Today, Digitimes | **0** |
| 科技 | TechCrunch, The Verge, Wired, 36氪 | **1** |
| 全球热点 | BBC, Al Jazeera, France24, DW, NHK | **0** |

**后果**：
- 身处中国半导体产业的员工，看不到中文世界的一手行业新闻
- 缺失：芯通社、半导体行业观察、集微网、天天IC、摩尔精英等核心中文源
- 缺失：中半协(CSIA)、深圳半导体行业协会、中国电子报等官方源
- 中国台湾的 Digitimes 只有 RSS 需要特殊代理，覆盖有限

**建议**：

推荐的**高价值中文 RSS 源**（需验证可抓取性）：

```
✅ 中文科技产业：
- 半导体行业观察 (https://www.semiinsights.com/feed) — 需验证
- 集微网 — 需确认 RSS 地址
- 芯通社 — 需确认 RSS 地址

✅ 行业协会/官方：
- 深圳市半导体行业协会 — 需确认 RSS
- 中国半导体行业协会 (CSIA) — 需确认 RSS

✅ 汽车芯片：
- 高工智能汽车 — 需确认 RSS
- 新智驾 — 需确认 RSS
```

#### P1-3：页面间信息孤岛 — 首页和大屏互不连通

**现状**：
- `index.html`（主页 5 Tabs）和 `bigscreen.html`（大屏）是两个完全独立的页面
- 主页没有大屏入口
- 大屏没有回主页的入口
- 大屏的实时行情（东财 JSONP）主页用不了

**建议**：
- 主页 Header 增加「📺 监控大屏」按钮，点击跳转到 `bigscreen.html`
- 大屏 Logo 区域增加「← 返回首页」按钮
- 大屏的实时行情数据同步给主页使用（共享 dataService）

#### P1-4：缺少"今日要闻"摘要视图

**现状**：新闻流是滚动列表，无人工编辑摘要，无重点突出

**后果**：
- 员工每天打开后面对 20+ 条新闻，不知道该看哪条
- 没有任何"头条"、"重要"、"最新"的视觉区分
- 缺少"今天最重要的 3 件事"这类高管摘要

**建议**：
- 增加「今日头条」区域（3 条最重要的新闻，手动或算法精选）
- 新闻按重要性（impact/heat）排序而非按时间排序
- 增加"过去 24 小时关键动态"时间线视图

#### P1-5：无通知/预警机制

**现状**：员工必须**主动打开网页**才能看到最新动态

**后果**：
- 如果供应商突然断供、竞争对手发布关键产品、政府出台重大政策，员工无法及时获知
- 作为监控系统，这是核心功能缺失

**建议**（逐步实施）：
1. **浏览器通知**（Web Notification API）：关键预警可推送到桌面
2. **邮件摘要**：每日/每周自动生成摘要邮件（需要后端服务）
3. **企业微信 Bot**：将关键预警推送到企业微信群

#### P1-6：无用户偏好设置

**现状**：每个员工看到的内容完全一样

**建议**（可选，长期）：
- 员工可以选择关注特定领域/公司/话题
- 首页可以根据用户偏好调整内容优先级
- 使用 `localStorage` 存储偏好，无需后端

### 🟢 P2 — 一般问题（影响体验但不致命）

#### P2-1：搜索只搜标题，不能搜全文/摘要

**现状**：搜索框 `newsSearchQuery` 只匹配新闻标题

**建议**：扩展搜索范围到 `summary`、`source`、`category`

#### P2-2：无日期范围筛选

**现状**：所有新闻混在一起，无法只看"今天"或"本周"

**建议**：增加日期快速筛选按钮（今天 / 本周 / 本月）

#### P2-3：无导出功能

**现状**：无法导出新闻列表/摘要报告

**建议**：增加"导出当前视图 → PDF/CSV"按钮（PDF 用 `window.print()`，CSV 用 Blob 下载）

#### P2-4：网页加载慢（Google Fonts + D3 + TopoJSON）

- Google Fonts 在中国加载慢（已在技术审查中指出）
- TopoJSON `world-110m.json` 约 430KB，首次加载慢
- 没有骨架屏以外的加载进度指示

**建议**：
- 切换 Google Fonts → 国内 CDN
- TopoJSON 预加载并缓存（已有 Service Worker，但需要优化配置）
- 加载进度条

#### P2-5：Mobile 完全不支持

**现状**：
- 页面在手机上完全不可用
- Header 5 个 Tab 按钮横向溢出
- 地图无法交互
- 无响应式设计

**建议**：至少为手机提供简化的"新闻列表 + 关键指数"视图

#### P2-6：公司 Logo 缺失

**现状**：
- 首页 Logo 是 CSS 圆圈 `◉`，不是真正的 Oritek Logo
- 大屏使用了 `oritek-logo.png` 但可能路径问题导致不显示

**建议**：统一使用 Oritek 品牌 Logo

## 五、与上次技术审查报告的交叉对比

上次技术审查（`REVIEW_REPORT.md`）关注的是 **代码质量、架构、视觉**。

本次实用性审查关注的是 **内容真实性、信息价值、用户体验**。

### 两报告互补的关键发现

| 技术审查问题 | 实用性后果 |
|------------|-----------|
| API Key 硬编码 | 不影响员工使用，但安全风险需要修复 |
| TypeScript 严格模式关闭 | 间接影响：bug 可能导致数据显示错误 |
| 无本地缓存 | **直接影响**：网络断开后，兜底假数据上场 |
| 静态图表（三角函数生成） | **致命影响**：用户一眼看穿，丧失信任 |
| 单文件过大 | 间接影响：扩展功能困难 |
| Google Fonts 慢 | **直接影响**：中国员工加载体验差 |
